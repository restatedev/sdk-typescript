New package: `@restatedev/restate-sdk-tunnel` — serve a Restate SDK deployment over an **outbound** connection to Restate Cloud's tunnel, with no inbound HTTP listener. `connectTunnel({ region | tunnelServers, environmentId, authToken, signingPublicKey, tunnelName, services })` dials the tunnel server (TLS with ALPN `h2`, the same offer the standalone Rust client makes; requires a tunnel server that advertises it — standard-h2 control traffic), role-flips so Restate Cloud drives HTTP/2 as the client while the deployment serves, completes the `/_/start-tunnel` credentials/trailers handshake, and dispatches each forwarded invocation into the SDK's own endpoint handler (full-duplex `BIDI_STREAM`) after stripping the tunnel's destination path prefix.

Request identity is enforced by delegation to the SDK: `signingPublicKey` (required) is passed to the endpoint handler, so every forwarded request's `x-restate-jwt-v1` is verified against the service-relative path your environment signed. Reconnects use jittered exponential backoff that resets only after a stable connection; authorization failures (`unauthorized`, `bad-tunnel-name`, tunnel-name mismatch) are fatal and surface on `connection.error` / `connection.ready` rather than retry-looping. Graceful drain is supported and advertised by default (`supportsDrain`): on `/_/drain-tunnel` the engine immediately dials a replacement connection while the old one keeps serving its in-flight invocations (bounded by `drainGraceMs`, default 120s) — zero dropped requests across cloud rollovers. The returned `TunnelConnection` exposes `ready`, `close()`, `connectionCount`, the handshake-learned `tunnelName` / `proxyUrl` / `tunnelUrl`, and a ready-made `deploymentUrl` — `<proxyUrl>/http/in-process/9080/` — to register with `restate dep register`. `tunnelName` is the deployment's identity: the tunnel server routes and load-balances purely by `<environment>/<tunnelName>`, so it must be unique per deployment and shared across that deployment's replicas; the `/http/in-process/9080/` destination segment is a constant that is never dialed for an in-process tunnel. Multi-homed like the standalone client: one connection per resolved tunnel server (SRV discovery — via `region` or a raw `tunnelServersSrv` name — expands every target to all of its addresses; explicit `tunnelServers` get one connection per entry), each with an independent reconnect loop, with the server set re-resolved every `resolveIntervalMs` (30s default) — servers that appear get connections, servers that vanish are torn down, and resolver blips (transport errors) keep existing connections serving rather than tearing them down.

Configuration falls back to `RESTATE_INPROC_*` environment variables when options are omitted (option > environment > throw): `tunnelName` ← `RESTATE_INPROC_TUNNEL_NAME`, `environmentId` ← `RESTATE_INPROC_ENVIRONMENT_ID`, `region` ← `RESTATE_INPROC_CLOUD_REGION`, `signingPublicKey` ← `RESTATE_INPROC_SIGNING_PUBLIC_KEY`, and `authToken` ← the file named by `RESTATE_INPROC_AUTH_TOKEN_FILE`, which is re-read on every reconnect so rotated Secrets are picked up without a restart. These form the contract with the restate-operator's `tunnelMode: in-process`, which injects the identity variables per revision and auto-registers the matching tunnel URL — there, `connectTunnel({ services })` plus a token-file Secret mount (named by `RESTATE_INPROC_AUTH_TOKEN_FILE`) is a complete configuration.
